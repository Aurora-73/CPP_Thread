## 1. 线程与互斥锁

**文件：1.cpp**

### std::thread

`std::thread` 表示一个执行线程，用于并发执行函数。

- **参数传递**：参数默认按值拷贝，即使函数参数是引用类型，也需要 `std::ref` 显式传引用
- **生命周期**：引用参数所引用对象的生命周期必须覆盖线程使用周期
- **移动语义**：支持移动，不支持拷贝
- **析构要求**：若 `std::thread` 析构时 `joinable() == true`，会调用 `std::terminate` 终止程序
- **C++20 改进**：`std::jthread` 实现析构时自动 join，还支持 `stop_token` 作为停止请求信号

**常用函数：**

| 函数                         | 说明                                |
| ---------------------------- | ----------------------------------- |
| `joinable()`               | 检查是否关联线程执行对象            |
| `get_id()`                 | 返回线程 id                         |
| `join()`                   | 等待线程执行结束                    |
| `detach()`                 | 线程独立运行，thread 对象失去控制权 |
| `this_thread::sleep_for()` | 当前线程休眠指定时长                |
| `this_thread::yield()`     | 让出 CPU 时间片，允许其他线程运行   |

### 互斥量类型

| 类型                           | 特点                                                                   |
| ------------------------------ | ---------------------------------------------------------------------- |
| `std::mutex`                 | 基础互斥量，同一时刻仅允许一个线程持有。同一线程重复 lock 是未定义行为 |
| `std::timed_mutex`           | 支持超时加锁：`try_lock_for`、`try_lock_until`                     |
| `std::recursive_mutex`       | 支持同一线程重复加锁，每次 unlock 释放一层持有                         |
| `std::recursive_timed_mutex` | recursive_mutex + timed_mutex                                          |
| `std::shared_mutex`          | 支持共享锁（读锁）和独占锁（写锁）                                     |

### 数据竞争与同步

- **数据竞争**：多个线程同时读写同一内存区域，结果是未定义行为
- **解决方案**：使用 mutex 保护共享数据
- **无竞争场景**：每个线程只访问独立的内存区域（如 vector 的不同下标）

## 2. RAII 锁管理

**文件：2.cpp**

RAII（Resource Acquisition Is Initialization）模式将锁的生命周期绑定到对象的生命周期，避免忘记解锁。

### lock_guard

严格基于作用域的互斥体所有权包装器，最简单、开销最小。

```cpp
template<class Mutex>
class lock_guard {
    Mutex& m;
public:
    explicit lock_guard(Mutex& m_) : m(m_) { m.lock(); }
    ~lock_guard() { m.unlock(); }
};
```

- 构造时加锁，析构时解锁
- 不支持移动和拷贝
- 可使用 `adopt_lock` 标签接管已持有的锁

**常见陷阱**：`lock_guard<mutex> {mtx};` 创建临时对象，表达式结束后立即析构，后续代码实际上无锁运行。

### unique_lock

可移动的互斥体所有权包装器，功能最丰富。

- 概念上包含两个成员：`mutex_type *pmutex; bool owns;`
- 支持多种构造策略：

| 标签            | 行为                               |
| --------------- | ---------------------------------- |
| 默认            | 立即加锁，owns = true              |
| `defer_lock`  | 不加锁，owns = false               |
| `try_to_lock` | 尝试加锁，owns = try_lock() 的结果 |
| `adopt_lock`  | 接管已持有的锁，owns = true        |

- 支持移动构造和移动赋值，不支持拷贝
- `release()`：放弃所有权但不解锁，返回 mutex 指针
- `lock()` / `try_lock()` / `unlock()`：手动管理锁状态

### scoped_lock

用于多个互斥体的死锁避免 RAII 包装器，内部使用死锁避免算法（如同 `std::lock`）。

### shared_lock

类似于 `unique_lock`，但操作的是共享锁（读锁），允许多个线程同时持有。

### 辅助函数

- `std::try_lock(lock1, lock2, ...)`：按顺序尝试加锁，失败则回退已加的锁
- `std::lock(lock1, lock2, ...)`：使用死锁避免算法同时锁定多个互斥体

## 3. call_once 与单例模式

**文件：3.cpp**

### std::call_once

保证某段代码在多线程环境下只执行一次。

```cpp
template<class Callable, class... Args>
void call_once(std::once_flag& flag, Callable&& f, Args&&... args);
```

**特性：**

1. 第一次调用时立即执行 f，后续调用直接返回
2. 若 f 抛出异常，once_flag 不会被标记为已调用，下次仍会尝试执行
3. 参数不会被移动或复制（在当前线程立即执行）

### 单例模式实现

**方式一：call_once + unique_ptr**

```cpp
class Log {
    static std::unique_ptr<Log> log;
    static std::once_flag once;
public:
    static Log& GetInstance() {
        std::call_once(once, [] { log.reset(new Log); });
        return *log;
    }
};
```

**方式二：static 局部变量（推荐）**

```cpp
class Log2 {
public:
    static Log2& GetInstance() {
        static Log2 instance;  // C++11 保证线程安全
        return instance;
    }
};
```

第二种方式更简洁，线程安全由 C++11 标准保证。

## 4. 条件变量

**文件：4.cpp**

条件变量是一种同步原语，允许多个线程相互通信。它总是与互斥体关联。

### condition_variable

只支持与 `std::unique_lock<mutex>` 关联。

**核心机制：**

```
wait() 的执行过程（原子性）：
1. 调用 lock.unlock()
2. 把当前线程挂到等待队列，阻塞线程
---
被唤醒后：
3. 调用 lock.lock()（可能阻塞）
4. wait() 返回，线程继续执行
```

**关键点：**

- `condition_variable` 不保存状态，只负责"通知变化发生过"，维护挂起队列
- 唤醒不意味着立即执行，线程还必须抢占 mutex 才能继续
- 通知线程不需要持有与等待线程相同的互斥锁

**带谓词的 wait：**

```cpp
void wait(unique_lock<mutex>& lock, Predicate p);
// 等价于：while (!p()) wait(lock);
```

### condition_variable_any

支持任意锁类型（如 `shared_lock`），可用于共享所有权模式下的等待。

### C++20 stop_token

```cpp
bool wait(Lock& lock, std::stop_token stoken, Predicate pred);
```

允许等待中的线程被"主动取消"，解决传统 CV 没有外部取消机制的问题。

### 虚假唤醒与丢失唤醒

**虚假唤醒（Spurious Wakeup）：**

- 线程被唤醒但没有收到 notify，条件可能仍不满足
- 原因：操作系统调度或底层实现机制（如 pthreads）
- 解决：必须循环检查条件

**丢失唤醒（Lost Wakeup）：**

- notify 发出时没有线程在等待，但已有线程准备阻塞
- 原因：没有互斥地访问条件
- 解决：用 mutex 保护条件，保证 wait 前的条件判断和 notify 前的条件修改互斥进行

### 最佳实践

```cpp
// 推荐：先 unlock 再 notify，减少虚假唤醒
{
    lock_guard<mutex> lk(mtx);
    pred = true;
}
cv.notify_all();
```

## 5. 生产者-消费者问题

**文件：5.cpp（全局变量版）、6.cpp（类封装版）**

### 问题描述

生产者负责生成数据存入缓冲区，消费者从缓冲区取出数据消耗。需要确保：

- 缓冲区满时生产者停止写入
- 缓冲区空时消费者停止读取

### 设计要点

1. **两个条件变量**：

   - `cv_not_full`：生产者等待"队列未满"
   - `cv_not_empty`：消费者等待"队列非空"
2. **同一把 mutex**：保护任务队列及相关状态
3. **退出机制**：

   - main 线程设置 `stop_produce`，表示不再接收新任务
   - 最后一个生产者退出时唤醒所有消费者
   - 消费者处理完剩余任务后退出
4. **wait 必须带谓词**：防止虚假唤醒

### TaskQueue 类（6.cpp）

```cpp
class TaskQueue {
    mutex mtx_;
    condition_variable cv_not_full_, cv_not_empty_;
    queue<string> tasks_;
    const int max_size_;
    bool stop_produce_ = false;
    int active_producers_;
};
```

## 6. Promise 与 Future

**文件：7.cpp**

Promise 和 Future 通过共享状态（shared_state）关联，共享状态保存异常指针、结果值和状态信息。

### promise

存储用于异步检索的值。

| 特性                           | 说明                                           |
| ------------------------------ | ---------------------------------------------- |
| `set_value()`                | 存储值并使状态就绪                             |
| `set_exception()`            | 存储异常并使状态就绪                           |
| `set_value_at_thread_exit()` | 线程退出后才使状态就绪                         |
| 析构行为                       | 若共享状态未就绪，存储 `broken_promise` 异常 |

### future

等待异步设置的值。

| 特性                    | 说明                                    |
| ----------------------- | --------------------------------------- |
| `valid()`             | 检查是否拥有共享状态                    |
| `get()`               | 阻塞等待结果，调用后 valid() 变为 false |
| `wait()`              | 阻塞等待但不提取结果                    |
| `wait_for/wait_until` | 带超时的等待                            |
| `share()`             | 转换为 shared_future                    |
| 析构行为                | 若是最后一个引用，销毁共享状态          |

### shared_future

支持多个对象共享同一个共享状态，调用 `get()` 后仍然有效，支持拷贝。

## 7. packaged_task 与 async

**文件：8.cpp**

### packaged_task

封装一个函数，以异步检索其返回值。

- 将函数返回值包装为 future 变量
- 调用函数后设置 future 的值
- 类本身不负责并发，只负责接入 future 同步模型

### std::async

异步运行函数并返回 `std::future`。

**启动策略：**

| 策略                 | 是否创建线程 | 任务何时运行       |
| -------------------- | ------------ | ------------------ |
| `launch::async`    | 是           | async 调用时       |
| `launch::deferred` | 否           | 第一次 wait/get 时 |

**重要特性：**

- 从 `std::async` 获得的 future，若未被移动或绑定到引用，析构时会阻塞到计算完成
- 从 `packaged_task`/`promise` 获得的 future，析构时不会阻塞

## 8. 线程池

**文件：9.cpp、9p.cpp、10.cpp**

### 9.cpp：固定任务签名线程池

- 队列中直接保存 `packaged_task<R()>`
- 每个实例只处理一种返回值类型
- 有界队列：满时等待

### 9p.cpp：继承式泛型线程池

- 使用继承实现类型擦除
- 先构造 `my::packaged_task<R()>` 保留 future
- 再包装成 `my::packaged_task<void()>` 统一放入队列

### 10.cpp：函数式泛型线程池

- 使用 `std::function<void()>` 或 `std::move_only_function<void()>`
- `shared_ptr` 托管 `packaged_task`（因为 `std::function` 需要可拷贝）
- C++23 下优先使用 `move_only_function`

### 线程池核心设计

```cpp
class ThreadPool {
    mutex mtx;
    condition_variable cv;
    atomic<bool> stop_flag = false;
    vector<thread> workers;
    queue<TaskType> tasks;

    void worker_loop() {
        while (true) {
            unique_lock<mutex> lk(mtx);
            cv.wait(lk, [&] { return stop_flag || !tasks.empty(); });
            if (stop_flag && tasks.empty()) break;
            auto task = std::move(tasks.front());
            tasks.pop();
            lk.unlock();
            task();
        }
    }
};
```

## 9. 原子操作与内存序

**文件：11.cpp**

### std::atomic

定义原子类型，保证对原子对象的访问可以建立线程间同步。

**类型要求：** 可平凡复制（TriviallyCopyable）、支持拷贝和移动、不含 const/volatile

### 核心操作

| 操作                             | 说明               |
| -------------------------------- | ------------------ |
| `store(desired, order)`        | 原子写入           |
| `load(order)`                  | 原子读取           |
| `exchange(desired, order)`     | 原子替换并返回旧值 |
| `compare_exchange_weak/strong` | CAS 操作           |
| `fetch_add/sub/and/or/xor`     | 原子读-修改-写     |

### compare_exchange

逐位比较（不使用 operator==），相等时替换，不等时加载实际值到 expected。

- **strong**：只有真正不相等才失败
- **weak**：即使相等也可能失败（CPU 架构限制），通常需要放在循环中使用

### 内存序

| 内存序      | 说明                                                      |
| ----------- | --------------------------------------------------------- |
| `seq_cst` | 默认，最强，所有操作形成单一全局顺序                      |
| `relaxed` | 仅保证原子性，不做额外约束                                |
| `acquire` | 获得语义，之后的读写不能重排到之前（类比 mutex.lock()）   |
| `release` | 释放语义，之前的读写不能重排到之后（类比 mutex.unlock()） |
| `acq_rel` | 同时要求读取和写入一致性                                  |

### atomic_flag

保证免锁的原子布尔类型，不提供 load/store，只提供 test_and_set/clear。

### 自旋锁实现

```cpp
// 标准自旋锁（CPU 空转）
class SpinLock {
    atomic_flag flag = ATOMIC_FLAG_INIT;
    void lock() { while (flag.test_and_set(memory_order_acquire)); }
    void unlock() { flag.clear(memory_order_release); }
};

// 带阻塞的自旋锁（类似 futex）
class Mutex {
    atomic_flag flag = ATOMIC_FLAG_INIT;
    void lock() {
        while (flag.test_and_set(memory_order_acquire))
            flag.wait(true);  // 阻塞等待
    }
    void unlock() {
        flag.clear(memory_order_release);
        flag.notify_one();
    }
};
```

## 10. 无锁数据结构

**文件：11.cpp、12.cpp**

### 无锁栈（LockFreeStack）

基于 `atomic<shared_ptr<Node>>` 实现。

- **ABA 保护**：`atomic<shared_ptr>` 的引用计数天然防御 ABA（节点不会在持有引用时被释放复用）
- **析构安全**：迭代断链，防止递归析构导致栈溢出

```cpp
void push(T value) {
    auto new_node = make_shared<Node>(std::move(value));
    new_node->next = head.load();
    while (!head.compare_exchange_weak(new_node->next, new_node));
}

optional<T> pop() {
    shared_ptr<Node> old_head = head.load();
    while (old_head && !head.compare_exchange_weak(old_head, old_head->next));
    if (!old_head) return nullopt;
    return std::move(old_head->data);
}
```

### Michael-Scott 无锁队列（LockFreeQueue）

基于 `atomic<shared_ptr<Node>>` 实现的无锁队列。

- **无锁**：所有操作均无阻塞，基于 CAS 循环
- **无 ABA**：引用计数天然防御
- **帮助推进**：当 tail 落后时，其他线程会帮助推进

### SPSC 有界无锁队列（SPSCRingQueue）

单生产者-单消费者的环形缓冲区。

- **无 CAS**：纯 atomic load/store，真正的 wait-free
- **性能**：O(1)，单次操作仅几个原子指令
- **容量**：上取整为 2 的幂，用位掩码替代取模

**内存序选择：**

```
生产者：
  tail.load(relaxed)   // 只有自己写，无需同步
  head.load(acquire)   // 同步消费者的 release
  tail.store(release)  // 发布数据

消费者：
  head.load(relaxed)   // 只有自己写，无需同步
  tail.load(acquire)   // 同步生产者的 release
  head.store(release)  // 释放槽位
```
